Music Object Version 3.2.0
RELEASE DATE: 1999-07-20

If you are upgrading from Music Object version 2.x and would like to know about the significant changes that have occurred between versions 2.x and 3.0.x, please consult the version history for Music Object Version 3.0.0.

What's New In This Version

Several noteworthy changes have been made to the Music Object API between versions 3.1.x and 3.2.x, some of which may require that your existing sonified code be modified in some small way. For guidelines on updating your code to be compliant with Music Object 3.2.x, consult the section Migrating Content from Music Object 3.1.x to 3.2.x.

NEW FEATURE - Core Components
Starting from version 3.2.0, the Music Object API is now available in three core components that allow authors to create faster loading pages in cases where only certain capabilities are needed. To find out more about using core components, consult the section Music Object Core Components.
NEW FEATURE - Language Modules Now Available
It is now possible to customize the prompts that pop up as a result of the user not having the Beatnik Player installed, so that they appear in a language appropriate to the Web content you are sonifying. For more information on using language modules, see the section Music Object Language Modules.
NEW FEATURE - myMusicObject = new Music ()
The more conventional syntax for the creation of new object instances is now also supported by Music Object 3.2.0 (and higher). Whereas, in previous versions it was always necessary to pass the name of the Music Object instance one wished to create as a parameter of the Music Object constructor, it is now no longer necessary to do this. As an example, the statement new Music ('myMusicObject') could now also be expressed as myMusicObject = new Music (). For backward compatibility reasons, the old form will continue to be supported.
NEW STATIC PROPERTY - Music.installerOptions
The new Music.installerOptions static property lets you customize certain aspects of the behavior of the Beatnik Player installer. A key way that this property can be used is to specify a Web URL that should be returned to after the Beatnik Player install process has successfully been completed.
NEW INSTANCE PROPERTY - locked
The new locked instance property lets Music Object instances be locked so that they cannot respond to methods that may change their state. This is useful for temporarily disabling page sonification events, since the locked state will obstruct attempts to play Groovoids, music files, or musical notes through locked instances.
API CHANGE - Debug Capabilities Now an Extension
The debug capabilities of the Music Object API, as explained in the section Debugging Aids, are now no longer a part of the core Music Object API. These capabilities are still available by using the new Music Object Extension - Debug.
API CHANGE - "engageAudio" Becomes a Static Method
The engageAudio method is no longer a method of individual Music Object instances, but is now only accessible as a static method of the Music Object class. It is now no longer correct to use a statement such as "myMusicObject.engage ()", but the statement "Music.engageAudio ()" should be used in its place. For guidelines on updating your sonified content for this change, consult the section Migrating Content from Music Object 3.1.x to 3.2.x.
API CHANGE - "getReverbType" Becomes a Static Method
The getReverbType method is no longer a method of individual Music Object instances, but should instead be used as a static method of the Music Object class. As an example, the statement "myMusicObject.getReverbType ()" should now be replaced with "Music.getReverbType ()". For guidelines on updating your sonified content for this change, consult the section Migrating Content from Music Object 3.1.x to 3.2.x.
API CHANGE - "setGlobalMute" Becomes a Static Method
The setGlobalMute method is no longer a method of individual Music Object instances, but should instead be used as a static method of the Music Object class. As an example, the statement "myMusicObject.setGlobalMute ()" should now be replaced with "Music.setGlobalMute ()". For guidelines on updating your sonified content for this change, consult the section Migrating Content from Music Object 3.1.x to 3.2.x.
API CHANGE - "setReverbType" Becomes a Static Method
The setReverbType method is no longer a method of individual Music Object instances, but should instead be used as a static method of the Music Object class. As an example, the statement "myMusicObject.setReverbType ()" should now be replaced with "Music.setReverbType ()". For guidelines on updating your sonified content for this change, consult the section Migrating Content from Music Object 3.1.x to 3.2.x.
BUG FIX - Extensions
A problem was found with the implementation of Music Object extensions which caused the extensions to not work on Netscape Navigator 3.x. This has been fixed so that extensions are now compatible even with Navigator 3.x versions.
IMPROVEMENT - Behavior of Music.enabled Static Property Extended
The behavior of the Music.enabled static property has now been enhanced to allow the property to be set to false even after player instances have been embedded and have begun playing. When set to false, this property can now act as a master switch to obstruct attempts to change the state of any instances on a page, thus intercepting attempts to play Groovoids, music files, or musical notes.
IMPROVEMENT - Compatibility Prompts Get New Look
The look for Music Object compatibility dialogs has been updated to more closely resemble system dialog boxes. This applies also to dialogs that are opened using the Music.promptUser static method.
IMPROVEMENT - Compatibility Prompts Now Centered
All Music Object compatibility prompts that pop up are now centered within the screen, making them immediately easier to read. This applies also to prompts that are opened using the Music.promptUser static method.
OPTIMIZATIONS
Some general code optimizations have been made since version 3.1.0, making certain processes a little more efficient.
