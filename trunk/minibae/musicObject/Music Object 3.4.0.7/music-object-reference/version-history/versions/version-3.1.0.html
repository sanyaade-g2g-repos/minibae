Music Object Version 3.1.0
RELEASE DATE: 1999-05-17

If you are upgrading from Music Object version 2.x and would like to know about the significant changes that have occurred between versions 2.x and 3.0.x, please consult the version history for Music Object Version 3.0.0.

What's New In This Version

Several noteworthy changes have been made to the Music Object API between versions 3.0.x and 3.1.x, some of which may require that your existing sonified code be modified in some small way. For guidelines on updating your code to be compliant with Music Object 3.1.x, consult the section Migrating Content from Music Object 3.0.x to 3.1.x.

NEW FEATURE - Music Object Extensions
For the first time, the Music Object API is being extended through the use of modular JavaScript files that are geared towards different types of sonification needs (Beatnik Inc. has made a commitment to freezing the size of the core Music Object API's code). Using the Music Object extensions is easy - authors can source in specific modules as needed by the pages being sonified. For example, Music Object Extension - Groups is ideal for making remix interfaces easier to code. Find out all about Music Object Extensions.
API CHANGE - "stopAll" Becomes a Static Method
The stopAll method is no longer a method of individual Music Object instances, but is now only accessible as a static method of the Music Object class. It is now no longer correct to use a statement such as "myMusicObject.stopAll ()", but the statement "Music.stopAll ()" should be used in its place.
API CHANGE - "getNoteName" and "getNoteNumber" Become Exclusively Static Methods
The getNoteName and getNoteNumber methods, which could previously be used as methods of any created Music Object instance, can now only be used as static methods of the Music object. This change has been made to streamline the API and cut down a little on the code size. It was decided that, since these methods have no specific different meaning on a per-instance basis, they should only be available at a global level in the form of static methods.
API CHANGE - "playGroovoid" Method No Longer Supported
For some versions of the Music Object API, the playGroovoid method has been deprecated. This means that the method was no longer officially sanctioned but was still supported for backward compatibility. In order to keep the Music Object code clean and lean, this function has now been officially removed from the API. It is, of course, still possible to play Groovoids using the play method.
API CHANGE - "fadeFromTo" Method No Longer Supported
The "fadeFromTo" method has always had exactly the same behavior as the "fadeVolume" method. Because of this redundancy and in the interests of streamlining the Music Object API and reducing its code size, this method has been dropped.
API CHANGE - "isReady" Method Replaced by "ready" Instance Property
The isReady method is no longer supported. It is now possible to determine if a player is ready for scripting control simply by querying the ready property of the instance. For example, "myMusicObject.isReady ()" may now simply be replaced with "myMusicObject.ready". This simplifies the Music Object API a little and optimizes the code size, since all the isReady method was doing was returning the value of the ready instance property, anyway.
API CHANGE - "getVersion" Method Replaced by "Music.playerVersion" Static Property
It was decided that it does not make sense for getVersion to be a method of individual Music Object instances, since the value returned is always the same regardless of the instance used to call the method. Consequently, this instance method is no longer supported. There is no static getVersion method to replace the instance level method. Instead, one should use the Music.playerVersion static property, which returns the exact same value as the old getVersion method.
API CHANGE - "ignoreNotReady" No Longer Supported
It was decided that the Music.ignoreNotReady static property is not of enough value to occupy code space in the Music Object API. The default behavior has always been to not report attempts to execute scripting control over a Music Object instance that is not yet ready for such control. Since few authors may ever have used this property for code debugging, it was decided that the size reduction in the "music-object.js" file from removing support for the property would be of greater benefit to most authors.
API CHANGE - "musicObject" Name No Longer Supported
When Music Object version 3.0.0 was released, the constructor name "musicObject" was changed to just "Music". However, for reasons of backward compatibility and easing the transition for authors, the constructor name "musicObject" was still supported as well. This has now changed and it is no longer possible to create Music Object instances using the constructor name "musicObject". If you have been using a 2.x version of the Music Object API and have still not transitioned your sonified code over to the newer 3.0.x version, consult the section Migrating Content from Music Object 2.x to 3.0.x.
API CHANGE - "mo_installBeatnik" No Longer Supported
When Music Object version 3.0.0 was released, the global function "mo_installBeatnik" was replaced with the static method "Music.installPlayer". However, for backward compatibility reasons, the function name "mo_installBeatnik" was initially also maintained. This is no longer the case with Music Object 3.1.0, and if you are still using the function in your code, you will have to switch over to using the "Music.installPlayer" static method. If you have been using a 2.x version of the Music Object API and have still not transitioned your sonified code over to the newer 3.0.x version, consult the section Migrating Content from Music Object 2.x to 3.0.x.
BUG FIX - VOLUME=0
A bug was introduced with Music Object version 3.0 that wasn't present in versions 2.x. The bug involved instances of the player embedded with the VOLUME attribute set to 0. Such instances would not play at all. This bug has been fixed in version 3.1.0.
WORKAROUND REINSTATED - Navigator 4.04 Issue
A workaround that was implemented in Music Object 2.1.6 for a bug in Navigator 4.04 was subsequently dropped in Music Object 2.5.5, due to several releases of Navigator passing since Netscape fixed the bug and also due to the code size that it added to the Music Object API. However, due to concerns from Web content developers regarding compatibility with all versions of Navigator, this workaround has been reinstated in the form of the optional Music Object Patch - Navigator 4.04 Bug.
OPTIMIZATIONS
Some general code optimizations have been made since version 3.0.8, making certain processes a little more efficient.
